Gateway API on AWS EKS

Hiraya quick reference: what it is, who owns what, and where traffic goes.

One-sentence model: Gateway API is Kubernetes' standard routing contract; AWS Load Balancer Controller reads that contract and creates/configures an AWS ALB for this project.

Core objects

GatewayClassNames the implementation. Hiraya uses aws-alb with controller gateway.k8s.aws/alb.
GatewayThe shared entry point. Hiraya uses edge/public with HTTP and HTTPS listeners.
HTTPRouteApplication/admin route rules. Example: hiraya.noidilin.dev routes to vintage/frontend:3000.
AWS ALBThe actual cloud load balancer created by AWS Load Balancer Controller from Gateway API resources.

Hiraya request flow

Browser ↓ DNS lookup Route 53 ALIAS record ↓ AWS ALB: hiraya-dev-public ↓ TLS terminates here Gateway API: edge/public + HTTPRoute vintage/frontend ↓ HTTP backend traffic Kubernetes Service: vintage/frontend:3000 ↓ /api/* proxy frontend nginx ↓ Kubernetes Service: vintage/gateway:3001 ↓ Node/Express app gateway ↓ auth / product-service / orders / user-service

Useful commands

kubectl get gateway -A
kubectl describe gateway -n edge public
kubectl get httproute -A
kubectl describe httproute -n vintage frontend
kubectl logs -n kube-system deploy/aws-load-balancer-controller --tail=100
kubectl logs -n external-dns deploy/external-dns --tail=100
kubectl get svc -n vintage frontend gateway -o wide
Project invariant: frontend and gateway Services should remain ClusterIP. Public traffic enters through the shared Gateway/ALB, then frontend nginx proxies /api/* to the in-cluster app gateway.
