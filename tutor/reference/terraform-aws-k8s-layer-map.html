Terraform, AWS, Kubernetes, Helm, and CRDs

Hiraya layer map for deciding “does this belong to AWS or Kubernetes?”

Classifier rule: ask where the object is stored and reconciled. If it lives in the AWS API, it is AWS. If it lives in the Kubernetes API, it is Kubernetes. If it is a Kubernetes workload with AWS IAM permissions that changes AWS resources, it is a bridge/controller.

Terraform providers in Hiraya

AWS provider
Talks to AWS APIs. Creates VPC, subnets, EKS cluster, node group, IAM roles/policies, OIDC provider, ACM cert, Route 53 validation records.
Kubernetes provider
Talks to the EKS Kubernetes API server. Creates objects such as namespaces directly, for example kubernetes_namespace_v1.external_dns.
Helm provider
Also talks to the Kubernetes API server. Installs Helm charts, which render many Kubernetes manifests as one versioned helm_release.

Helm and CRDs

TermMeaningHiraya example
Helm chartA package of templated Kubernetes YAML.AWS Load Balancer Controller chart, ExternalDNS chart, local edge-gateway chart.
Helm releaseOne installed instance of a chart in a cluster.helm_release.controller, helm_release.external_dns, helm_release.edge_gateway.
CRDA CustomResourceDefinition that adds a new Kubernetes API kind.Gateway, HTTPRoute, LoadBalancerConfiguration.
Custom resourceAn object of a kind introduced by a CRD.Gateway edge/public, HTTPRoute vintage/frontend.
ControllerA loop that watches Kubernetes objects and changes reality to match.AWS LBC watches Gateway resources and creates ALB resources.

Review commands

# AWS-side evidence
terraform state list | grep '^module.vpc\|^module.eks\|aws_iam\|aws_acm\|aws_route53'
aws elbv2 describe-load-balancers
aws route53 list-resource-record-sets --hosted-zone-id ZONE_ID

# Kubernetes-side evidence
kubectl get ns
kubectl get crd | grep -E 'gateway|externaldns|monitoring'
kubectl get gateway,httproute -A
kubectl get deploy,sa -n kube-system aws-load-balancer-controller
kubectl get deploy,sa -n external-dns external-dns
