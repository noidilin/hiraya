Trace Gateway API to AWS ALB

Lesson 0001 · A five-minute map for the Hiraya EKS edge.

Tangible win: after this lesson, you should be able to say which component handles each part of public traffic: Kubernetes routing intent, AWS load balancer creation, DNS, frontend routing, and backend API aggregation.

1. The smallest correct model

Gateway API is Kubernetes' role-oriented API for L4/L7 service networking. It gives you resources such as GatewayClass, Gateway, and HTTPRoute. Kubernetes itself does not magically create an AWS ALB from these objects; an implementation must reconcile them.

In Hiraya, that implementation is AWS Load Balancer Controller. Its Gateway API support maps L7 HTTPRoute/GRPCRoute traffic to an AWS ALB.

1. Terraform installs Gateway API CRDs, AWS Load Balancer Controller, ExternalDNS, and the shared edge/public Gateway.
↓
2. GitOps applies the app HTTPRoute: hiraya.noidilin.dev → vintage/frontend:3000.
↓
3. AWS Load Balancer Controller reconciles Gateway API objects into an internet-facing AWS ALB.
↓
4. ExternalDNS reads HTTPRoute hostnames and writes Route 53 records pointing to the ALB.

2. Where the app gateway fits

The app's gateway microservice is not Gateway API. It is a Node/Express backend service that proxies app API paths such as /api/auth, /api/products, /api/orders, and /api/users to internal microservices.
