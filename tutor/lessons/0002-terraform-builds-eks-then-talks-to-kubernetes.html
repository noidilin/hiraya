Terraform Builds EKS, Then Talks to Kubernetes

Lesson 0002 · The building blocks behind Terraform-managed Kubernetes resources on EKS.

Tangible win: you will be able to point at a Terraform block and say whether it creates the EKS cluster itself, talks to the Kubernetes API, or installs a Helm chart into the cluster.

1. The two-stage model

Terraform can provision EKS with the AWS provider. But after the cluster exists, Terraform can also become a Kubernetes client by configuring the Kubernetes provider or the Helm provider with that cluster's endpoint, CA certificate, and auth token.

Stage A — AWS provider: create the container platform: VPC, EKS control plane, node group, IAM, OIDC, ACM, Route 53 validation.
↓
Stage B — Kubernetes/Helm providers: connect to the EKS Kubernetes API server and create in-cluster objects: namespaces, Helm releases, CRDs, controllers, Gateways.

In Hiraya, this connector appears in infra/envs/dev/platform/main.tf:

data "aws_eks_cluster_auth" "eks" {
  name = module.eks.cluster_name
}

provider "kubernetes" {
  alias                  = "eks"
  host                   = module.eks.cluster_endpoint
  cluster_ca_certificate = base64decode(module.eks.cluster_certificate_authority_data)
  token                  = data.aws_eks_cluster_auth.eks.token
}

provider "helm" {
  alias = "eks"
  kubernetes = {
    host                   = module.eks.cluster_endpoint
    cluster_ca_certificate = base64decode(module.eks.cluster_certificate_authority_data)
    token                  = data.aws_eks_cluster_auth.eks.token
  }
}

2. Helm: package manager for Kubernetes

Helm is the package manager for Kubernetes. A chart is a package of Kubernetes templates. A release is one installed instance of a chart. Terraform's helm_release resource describes the desired status of a chart in a Kubernetes cluster.

TerraformMeaningHiraya example
chartWhat package to install.aws-load-balancer-controller or local ${path.module}/chart.
valuesConfiguration passed into chart templates.cluster name, VPC ID, service account annotation, Gateway name.
waitTerraform waits for readiness before considering release applied.Used for LBC, ExternalDNS, Gateway chart.
namespaceWhere chart resources are installed.kube-system, external-dns, edge.

3. CRD: teaching Kubernetes a new noun

A CustomResourceDefinition adds a new resource type to the Kubernetes API. After a CRD is installed, you can use kubectl get and apply YAML for that new kind, just like built-in resources.

Hiraya needs Gateway API CRDs before it can create Gateway and HTTPRoute objects. It also needs AWS Load Balancer Controller Gateway CRDs before it can use AWS-specific custom resources like LoadBalancerConfiguration. 

# Terraform installs CRDs via a local Helm chart
module "gateway_api_crds" {
  source = "../../../modules/gateway-api-crds"
  providers = { helm = helm.eks }
  depends_on = [module.eks]
}

# Later, another Helm release can create custom resources
resource "helm_release" "edge_gateway" {
  chart = "${path.module}/chart"
}

4. The build sequence in Hiraya

  1. module.vpc creates AWS networking.
  2. module.eks creates the managed Kubernetes cluster and nodes.
  3. data.aws_eks_cluster_auth.eks gets a token so Terraform can talk to the Kubernetes API.
  4. provider.kubernetes.eks and provider.helm.eks connect to that API.
  5. helm_release.gateway_api_crds installs new Kubernetes API types.
  6. helm_release.controller installs AWS Load Balancer Controller.
  7. helm_release.edge_gateway creates Gateway API resources that the controller can reconcile.

5. Retrieval practice

Use one word: aws, kubernetes, helm, crd, or token.
