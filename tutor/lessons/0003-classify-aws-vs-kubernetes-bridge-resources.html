Classify AWS vs Kubernetes Bridge Resources

Lesson 0003 · Understanding AWS LBC, ExternalDNS, and Terraform's mixed ownership.

Tangible win: you will learn a classifier for confusing integrations: “where does this object live, who watches it, and what external API does it change?”

1. The confusion is real

Terraform in Hiraya provisions both AWS infrastructure and Kubernetes resources. The confusing parts are integrations such as AWS Load Balancer Controller and ExternalDNS: they run in Kubernetes, but they change AWS.

AWS object
Stored in AWS APIs. Example: IAM role, ACM certificate, ALB, Route 53 record.
Kubernetes object
Stored in the Kubernetes API server. Example: Namespace, ServiceAccount, Deployment, Gateway, HTTPRoute.
Bridge controller
A Kubernetes workload with AWS credentials. It watches K8s objects and calls AWS APIs.

2. The classifier

For any resource, ask these three questions:

  1. Where is the object stored? AWS API or Kubernetes API?
  2. Who reconciles it? Terraform, Kubernetes controller, AWS Load Balancer Controller, ExternalDNS?
  3. What side effect does it cause? Create a pod? Create an ALB? Write DNS?

This prevents a common mistake: classifying an object by the word “AWS” in its name. LoadBalancerConfiguration sounds AWS-ish, but it is a Kubernetes custom resource. Its side effect is AWS-ish because AWS LBC reads it.

3. AWS Load Balancer Controller is a bridge

In Hiraya, Terraform creates both sides of the bridge:

Terraform objectLives inPurpose
aws_iam_role.controllerAWSLets the controller pod call ELB, EC2, ACM, WAF APIs.
helm_release.controllerKubernetesInstalls the controller Deployment and ServiceAccount into kube-system.
ServiceAccount annotationKubernetesPoints the pod to its AWS IAM role via IRSA.
Gateway / HTTPRouteKubernetesDesired routing model.
ALB / listeners / target groupsAWSRuntime resources created by the controller.
 
# AWS side: IAM role trusts one Kubernetes service account
"system:serviceaccount:kube-system:aws-load-balancer-controller"

# Kubernetes side: Helm chart creates that service account
serviceAccount:
  name: aws-load-balancer-controller
  annotations:
    eks.amazonaws.com/role-arn: arn:aws:iam::...:role/...-aws-lbc

IRSA is the identity bridge: AWS IAM trusts a specific Kubernetes service account through the EKS OIDC issuer. AWS docs describe this as associating an IAM role with a Kubernetes service account so pods using that service account can access AWS APIs.

4. ExternalDNS is the same pattern

ExternalDNS runs as a Kubernetes Deployment, but it needs AWS Route 53 permissions. In Hiraya, Terraform creates an IAM policy scoped to Route 53, creates an IRSA role, creates the external-dns namespace, and installs the Helm chart. 

ExternalDNS pod in Kubernetes
  watches HTTPRoute hostnames: hiraya.noidilin.dev
  assumes IAM role through ServiceAccount annotation
  calls Route 53 APIs
  writes ALIAS/TXT records

ExternalDNS Gateway API docs say Route annotations and hostnames drive DNS, and the AWS tutorial describes the Route 53 IAM permissions and IRSA method. So classify it as: Kubernetes workload, AWS side effects.
