# GitOps-owned Cluster Platform add-ons

Status: accepted

Hiraya will split the current dev platform stack into Project Bootstrap, Platform Core, Cluster Bootstrap, Cluster Platform, and GitOps app ownership. Platform Core Terraform owns AWS and EKS-side foundations only, including VPC, EKS, EKS managed add-ons, ACM/DNS primitives, IAM/IRSA roles and policies, Secrets Manager resources, and non-secret outputs consumed by GitOps; it must not use Kubernetes or Helm providers. A separate Cluster Bootstrap Terraform state runs after Platform Core apply and installs Argo CD plus the app-of-apps root, while Argo CD owns one Application per Cluster Platform add-on, including Gateway API CRDs, AWS Load Balancer Controller, ExternalDNS, External Secrets Operator, edge Gateway resources, logging, monitoring, public admin routes, and workload app Applications.

This supersedes the shared-controller ownership model in ADR-0001 and refines ADR-0002 by keeping AWS/IAM/CloudWatch-side observability resources in Terraform while moving in-cluster collectors and add-on manifests to Argo CD. The dev migration will use new Terraform state keys and rebuild the disposable dev platform rather than doing Terraform Kubernetes state adoption. Argo CD will auto-sync from Git after CI validation; GitHub Actions will apply Platform Core, then Cluster Bootstrap as separate jobs/states, and will verify convergence through Kubernetes and public route smoke checks without calling the Argo CD API.

GitOps values should avoid dynamic AWS IDs that change on rebuilds. The edge Gateway will rely on AWS Load Balancer Controller VPC and ACM certificate discovery where possible; committed values may include stable names, domains, chart versions, and fixed IRSA role ARNs, but not VPC IDs or ACM certificate ARNs. Public Gateway Access is platform-granted: Cluster Platform owns public workload namespaces such as `vintage` and `monitoring`, while Cluster Bootstrap owns the special `argocd` namespace needed before Argo CD exists.

Platform admin secrets are generated by Terraform into AWS Secrets Manager and retrieved by Operators from Secrets Manager, not Terraform outputs. Project Bootstrap owns durable Vintage Storefront dev secrets; consumers own their ExternalSecret mappings, and ESO reads only allowlisted secret ARNs while GitOps manifests reference stable secret names. Public Argo CD and Grafana routes remain enabled for dev with strong generated credentials only; WAF, SSO, and IP allowlisting are deferred hardening work. HTTPRoute objects are owned by the service owner: Vintage owns its public app route, monitoring owns the Grafana route, and the Argo CD platform access app owns the Argo route; Edge owns only the shared Gateway, GatewayClass/configuration, and HTTP-to-HTTPS redirect.

Project Bootstrap creates a dedicated GitHub cluster-bootstrap role. Platform Core grants EKS admin only to the Operator's Dev SSO role and that bootstrap role; GitHub plan and core-apply roles do not receive Kubernetes API access. Destroy must suspend the root Argo app, prune child Applications in explicit order, keep AWS LBC and ExternalDNS running until managed ALB/DNS resources are gone, wait for Vintage PVC/EBS cleanup, then destroy Cluster Bootstrap and Platform Core with their separate roles.
